\section{Data composition}

The v14 model is fine-tuned jointly on the training splits of all three
benchmarks. GrailQA dominates the corpus because its training split is an order
of magnitude larger than WebQSP's. GrailQA's official test split is unlabelled
(leaderboard only), so its \emph{dev} split serves as the evaluation set, as is
common practice.

\subsection{Leakage check}
\label{sec:leakage}

Identifier intersection between the training corpus and each test set is empty
in all three cases. The check is not vacuous: CWQ derives its questions from
WebQuestions and reuses the base identifier with an appended hash
(\texttt{WebQTest-322\_bd42258\ldots}), so \num{8493} training items carry a
\texttt{WebQTest} prefix without being WebQSP test items.

\section{Training configuration}

The official MemQ repository prescribes only ``follow the LLaMA-Factory,
\texttt{lora rank=16}, \texttt{epoch=5.0}''; the paper itself reports no
fine-tuning hyperparameters. The v9 configuration is therefore the reference
for comparability. Sequence packing is the one deliberate deviation: without it,
per-batch padding produces varying GEMM shapes that intermittently trigger an
illegal memory access in a Tensile kernel on gfx1201, which terminates the run.
Packing fixes every batch at \(2 \times 2048\) tokens, which also removes a
large amount of padding waste.

\section{Checkpoint selection}

The run recorded no validation loss (\texttt{eval\_strategy: "no"}), so the
five per cent held-out split was scored retrospectively for seven candidates.
The earliest retained checkpoint is the best, and the final one is
\SI{5.8}{\percent} worse -- the same overfitting pattern the v11 run exhibits,
where the validation loss reaches its minimum at \SI{79.9}{\percent} of training
and then rises by \SI{16.8}{\percent}. Checkpoint~13750 corresponds to
\SI{79.1}{\percent} of training, so the optimum sits at the edge of the retained
range; earlier checkpoints had already been deleted by
\texttt{save\_total\_limit}.

\section{WebQSP: v14 against v9}

Both models were evaluated on the same \num{300} WebQSP test questions, with
identical memory-retrieval settings and against the same Freebase endpoint. The
v9 baseline reproduces its documented full-test-set score (\num{0.8423} versus
\num{0.8419} reported), which supports the representativeness of the subset.

\begin{table}[htbp]
  \centering

  structurally better queries but answers slightly less accurately.}

This reproduces the divergence already reported for v11, which improved
structure accuracy substantially while regressing answer-level metrics. Two
explanations are consistent with the data and cannot be separated here: the
DeepSeek-generated memory phrasing, and the reduced WebQSP share of the joint
corpus (Table~\ref{tab:share}). Note that v14 uses the same memory at training
and inference time, so train/inference consistency alone does not prevent the
effect.

With \(n = 300\) and \(p \approx 0.82\), the standard error is about
\SI{2.2}{pp}; the \SI{-4.0}{pp} difference in Hits@1 is therefore borderline,
whereas the \SI{+17.3}{pp} structural gain is far outside sampling noise.

\section{Pipeline defects found during evaluation}

Two defects were identified and fixed while producing the numbers above. Both
affected \emph{all} models evaluated with this pipeline, not only v14, and the
figures in Table~\ref{tab:webqsp} were obtained after the fixes.

\begin{table}[htbp]
  \centering
  \caption{Effect of the name-cache fix on v14 (cp13750), WebQSP, \num{300}
  questions. Merging GrailQA entity names had overwritten \num{451} established
  WebQSP/CWQ names with GrailQA's shorter \texttt{friendly\_name}
  (e.g.\ ``nixon'' for ``Richard Nixon''), so entity resolution failed for every
  affected question.}
  \label{tab:bugfix}
  \begin{tabular}{lS[table-format=1.4]S[table-format=1.4]}
    \toprule
    Metric & {Before fix} & {After fix} \\
    \midrule
    Reconstruction failures & 60 & 1 \\
    Structure accuracy      & 0.5033 & 0.6233 \\
    Hits@1                  & 0.6275 & 0.8020 \\
    Macro-F1                & 0.6273 & 0.7991 \\
    \bottomrule
  \end{tabular}
\end{table}

The second defect concerns whitespace: \SI{5.6}{\percent} of the DeepSeek memory
descriptions begin with whitespace, and the reconstruction step unconditionally
inserts a blank before every variable. Both effects produce double blanks, which
the plan parser rejects, so initially \emph{all} \num{300} reconstructions
failed. Collapsing repeated blanks before parsing resolves this without
affecting v9.
